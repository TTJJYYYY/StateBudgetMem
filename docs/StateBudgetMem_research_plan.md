# 资源受限端侧个人智能体的时态一致性长期记忆管理

> **StateBudgetMem: Temporally Consistent Long-Term Memory for Resource-Constrained On-Device Personal Agents**

**文档性质：** 研究路线与论文支撑  
**研究类型：** 端侧智能体长期记忆管理研究型项目  
**核心主线：** 时态一致性、版本感知、双视图记忆、查询相关有效性、预算约束检索与端侧评测

---

## 一、研究题目与核心目标

### 暂定题目

**资源受限端侧个人智能体的时态一致性长期记忆管理**

英文暂定：

**StateBudgetMem: Temporally Consistent Long-Term Memory for Resource-Constrained On-Device Personal Agents**

### 研究目标

研究一个长期运行的个人智能体，在用户信息不断变化、端侧存储空间有限、每次能够输入大模型的记忆 token 有限时，如何：

1. 保存用户当前有效状态；
2. 正确处理新旧信息的更新与冲突；
3. 避免错误使用已经过期的记忆；
4. 在需要时仍能回答历史问题；
5. 以尽可能低的存储、token 和检索延迟完成上述任务。

你们 PPT 原本提出的长期记忆写入、检索、更新、遗忘和用户控制，在这里被进一步收缩成一个具体科学问题：**动态用户状态下的记忆有效性管理与资源分配。**

---

## 二、研究问题从哪里产生？

普通长期记忆系统通常把对话提炼成若干文本，再放进向量数据库。例如：

```text
M1：用户不吃辣
M2：用户现在可以接受微辣
M3：用户最近胃不舒服，暂时不能吃辣
```

当用户问：

> 今天给我推荐一道菜。

三条记忆和“吃辣”都很相似，普通向量检索可能将它们同时取出。问题在于：

- 语义相似度无法判断哪条目前有效；
- 最新的记忆不一定永久替代旧记忆；
- 暂时状态和长期偏好不同；
- 旧信息对于当前问题无效，但对于历史问题仍然有价值；
- 端侧不能无限保留和输入所有历史。

因此，项目的核心矛盾是：

> **只保留最新状态，会失去历史；保留全部历史，又容易错误使用过期信息。**

[LongMemEval](https://openreview.net/forum?id=pZiyCaVuti) 已经把“时间推理、知识更新和信息不足时拒答”列为长期记忆系统的核心能力；[Memora](https://arxiv.org/abs/2604.20006) 的实验进一步表明，现有记忆智能体会频繁重复使用已经失效的信息。最新的 [STALE](https://arxiv.org/abs/2605.06527) 和 [MemConflict](https://arxiv.org/abs/2605.20926) 则分别指出，旧记忆可能被后续事件隐式失效，而且一条记忆是否有效还与当前问题有关。

---

## 三、总体研究假设

你们可以提出下面这个研究假设：

> 相比将所有记忆平等存入同一个向量库，将“当前用户状态”和“历史状态版本”分开管理，并在检索时考虑查询类型、记忆有效性、过期风险和 token 成本，能够在相同资源预算下减少过期记忆误用，同时保留历史时间推理能力。

整个路线围绕这个假设展开：

```text
普通扁平记忆
    ↓
加入结构化记忆抽取
    ↓
加入版本关系和当前状态
    ↓
分离当前状态视图与历史版本视图
    ↓
根据问题选择不同视图
    ↓
在端侧预算下选择记忆
    ↓
比较准确率、过期错误和资源消耗
```

## 四、研究路线一：建立一个可比较的基础记忆系统

第一步不是直接设计复杂算法，而是先搭建最基础的长期记忆实验框架：

```text
多轮对话
  ↓
提取值得保存的信息
  ↓
写入记忆库
  ↓
根据新问题检索相关记忆
  ↓
把记忆加入提示词
  ↓
生成答案并评测
```

### 这一阶段具体研究什么？

主要研究三个基础环节：

#### 1. 原子记忆抽取

把一段自然语言转化为单一、明确的记忆。

例如：

```text
原始输入：
我最近胃不舒服，医生说这段时间尽量不要吃辣。

原子记忆：
用户因胃部不适，当前应避免辣味食物。
```

#### 2. 结构化保存

每条记忆不只保存一段文字，还保存：

```text
主体：用户
属性：辣度偏好
取值：暂时避免吃辣
时间：2026-06-29
类型：临时健康限制
重要性：高
```

#### 3. 基础向量检索

先实现普通的语义 Top-K 检索，作为后续所有实验的基础基线。

### 论文支撑

[Mem0](https://arxiv.org/abs/2504.19413) 已经证明了“从持续对话中动态抽取、整合和检索关键记忆”比直接使用完整历史更高效，它可以作为这一阶段的主要系统参考。[MemoryBank](https://ojs.aaai.org/index.php/AAAI/article/view/29946) 则提供了长期保存、召回、更新和重要性管理的基础框架。

这一阶段不追求创新，目标是得到一个能够稳定批量运行的基线系统。

## 五、研究路线二：从“文本记忆”升级为“状态版本”

基础向量库把每条记忆视为彼此独立的文本，但用户信息实际上是会变化的状态。

你们需要将记忆改造成：

```text
用户属性 + 属性取值 + 有效时间 + 当前状态 + 版本关系
```

例如：

```text
M1
属性：辣度偏好
值：不吃辣
有效期：1月—3月
状态：历史

M2
属性：辣度偏好
值：可以接受微辣
有效期：3月—5月
状态：历史
替代：M1

M3
属性：饮食限制
值：暂时不能吃辣
有效期：5月至今
状态：当前
暂时使 M2 失效
```

### 需要定义的更新操作

新记忆到来后，系统执行以下一种操作：

| 操作              | 含义                   |
| ----------------- | ---------------------- |
| `ADD`             | 增加一个新的独立状态   |
| `MERGE`           | 补充已有记忆           |
| `SUPERSEDE`       | 新状态永久替代旧状态   |
| `TEMP_INVALIDATE` | 新状态暂时使旧状态失效 |
| `DELETE`          | 用户要求删除           |
| `NOOP`            | 重复、无关或不值得写入 |

例如：

```text
“我不吃辣”
→ ADD

“我现在可以接受微辣”
→ SUPERSEDE

“我最近胃不舒服，暂时不能吃辣”
→ TEMP_INVALIDATE
```

### 论文支撑

[A-MEM](https://openreview.net/forum?id=FiM0M8gcct) 表明记忆可以动态建立连接，并在新信息加入后更新已有记忆的表示，因此支持“记忆不是孤立文本块，而是可演化结构”这一设计。[STALE](https://arxiv.org/abs/2605.06527) 进一步表明，新事件可能在没有直接否定的情况下让旧状态失效；[MemConflict](https://arxiv.org/abs/2605.20926) 则把冲突区分为时间冲突、事实冲突和条件冲突，支持你们区分永久替代、临时失效和条件有效。

这一阶段的核心产出，是一个**版本感知的记忆更新模块**。

## 六、研究路线三：当前状态和历史版本分开管理

只增加版本字段还不够。如果检索时仍然把所有版本放进同一个候选池，旧状态依然可能与当前状态竞争。

因此建议设计两个记忆视图。

### 1. 当前状态视图

只保存用户目前最可能有效的信息：

```text
辣度状态：暂时避免吃辣
原因：胃部不适
生效时间：2026-05-10
置信度：0.91
```

它用于：

- 当前推荐；
- 当前计划；
- 个性化回答；
- 日常决策。

特点是规模小、检索快、不容易混入旧状态。

### 2. 历史版本视图

保存用户状态完整变化过程：

```text
不吃辣
→ 可以接受微辣
→ 因胃部不适暂时避免吃辣
```

它用于：

- 回答过去状态；
- 解释状态变化；
- 查询变化原因；
- 恢复暂时失效前的长期状态。

### 为什么需要双视图？

因为两个简单方案都有明显缺陷：

```text
只保留当前状态
→ 当前问题准确，但无法回答历史问题

所有历史统一检索
→ 历史保存完整，但容易误用旧信息
```

双视图的目标，是同时保留：

- 当前状态准确性；
- 历史时间推理能力。

### 论文支撑

[LongMemEval](https://openreview.net/forum?id=pZiyCaVuti) 同时评测知识更新和时间推理，说明记忆系统不能只保存当前事实，也必须理解历史变化。[MemConflict](https://arxiv.org/abs/2605.20926) 强调记忆有效性与查询相关：旧地址对于“现在住哪里”无效，但对于“以前住哪里”仍然有效。[A-MEM](https://openreview.net/forum?id=FiM0M8gcct) 的关联式记忆结构也为保存记忆演化关系提供了方法基础。

这一阶段是你们方法最重要的结构设计。

## 七、研究路线四：让检索理解用户在问“现在”还是“过去”

系统收到问题后，不应立刻在所有记忆中进行向量搜索，而应先判断问题类型。

建议划分为：

```text
CURRENT
询问当前状态或要求当前建议

HISTORICAL
询问过去某个时间的状态

CHANGE
询问状态发生了什么变化

GENERAL
不需要调用个人记忆
```

例如：

| 用户问题                 | 查询类型   | 应访问的记忆       |
| ------------------------ | ---------- | ------------------ |
| 我今天适合吃什么？       | CURRENT    | 当前状态视图       |
| 我去年是不是不吃辣？     | HISTORICAL | 历史版本视图       |
| 我的饮食习惯怎么变化的？ | CHANGE     | 当前＋历史版本     |
| 什么是低脂饮食？         | GENERAL    | 通常不需要个人记忆 |

这样可以避免一个常见错误：

> 因为旧记忆和问题语义相似，就把它当成当前事实。

### 查询相关的有效性

一条记忆不是永远有效或永远无效，而是要结合问题判断。

例如旧记忆：

```text
用户2025年住在北京。
```

对于：

> 用户现在住哪里？

它是无效证据。

对于：

> 用户2025年住哪里？

它是正确证据。

### 论文支撑

[MemConflict](https://arxiv.org/abs/2605.20926) 将记忆有效性定义为与查询相关的“是否适合使用”，并发现最终回答正确与检索排序正确并不总是一致。[LongMemEval](https://openreview.net/forum?id=pZiyCaVuti) 的时间推理与知识更新任务则可以直接用于检验查询分类和双视图检索是否有效。

## 八、研究路线五：加入过期风险感知的预算检索

到这里，你们已经解决了“哪条记忆当前有效”的问题。接下来才引入真正的端侧约束。

### 端侧的两个预算

#### 存储预算

设备只能保存有限的：

- 记忆文本；
- 元数据；
- 向量；
- 历史版本。

记作：

\[
B_s
\]

#### 上下文预算

每次只能向大模型输入有限数量的记忆 token。

记作：

\[
B_t
\]

因此不能简单地把所有有效记忆都输入模型。

### 记忆选择评分

可以给每条候选记忆计算：

\[
S(m,q)=
\alpha R(m,q)
+\beta V(m,q)
+\gamma I(m)
-\delta P_{\text{stale}}(m,q)
-\eta C(m)
\]

其中：

- \(R(m,q)\)：记忆与问题的语义相关性；
- \(V(m,q)\)：该记忆在当前问题下是否有效；
- \(I(m)\)：重要性；
- \(P_{\text{stale}}\)：使用过期记忆的风险；
- \(C(m)\)：记忆占用的 token 或字节成本。

随后在总成本不超过 \(B_t\) 的情况下，选择一组记忆。

你们真正要验证的不是公式中的某一组权重，而是：

> **在相同 token 预算下，加入有效性和过期风险后，是否比普通 Top-K 检索更准确？**

### 存储空间不足时怎么办？

可以按照记忆价值采取不同策略：

```text
当前关键状态
→ 完整保留

重要历史状态
→ 保留结构化版本

普通历史事件
→ 压缩为摘要

低价值、重复、已失效记忆
→ 归档或淘汰
```

### 论文支撑

[MemoryBank](https://ojs.aaai.org/index.php/AAAI/article/view/29946) 使用时间和重要性管理记忆，为时间衰减和重要性评分提供基础。端侧预印本 *[Forget to Improve](https://arxiv.org/abs/2606.25115)* 明确将端侧记忆视为受 RAM、能耗和通信约束的有限资源，并通过单位字节价值管理保留与淘汰。[DeMem](https://arxiv.org/abs/2605.10870) 则提出，预算下记忆的价值不应只由文本相似度决定，而应考虑忘掉它是否会改变后续决策。

需要注意，*[Forget to Improve](https://arxiv.org/abs/2606.25115)* 和 [DeMem](https://arxiv.org/abs/2605.10870) 是较新的 arXiv 预印本。你们可以把它们作为研究依据，但不应声称预算淘汰本身是你们首次提出的。

## 九、研究路线六：状态不确定时，不再默认使用旧记忆

还有一种情况：

```text
旧状态：用户可以接受微辣
新信息：用户最近胃不舒服
```

系统可以推断旧状态可能已经不适用，但新信息没有明确说明：

- 完全不能吃辣；
- 只能暂时少吃；
- 目前已经恢复。

这时系统不应该继续默认“可以吃微辣”，也不应该擅自确定“完全不能吃辣”。

可以引入：

```text
UNKNOWN_CURRENT
当前状态不确定
```

系统回答：

> 你之前可以接受微辣，但最近的胃部情况可能使这一偏好暂时失效。目前无法确认是否已经恢复，建议先选择清淡食物。

这体现了：

- 抵抗用户问题中的错误旧前提；
- 信息不足时拒绝武断回答；
- 将不确定性显式表示。

### 论文支撑

[STALE](https://arxiv.org/abs/2605.06527) 专门评测系统能否发现隐式失效、拒绝接受问题中的过时前提，并根据新状态调整行为。[LongMemEval](https://openreview.net/forum?id=pZiyCaVuti) 也把信息不足时的 abstention 作为核心长期记忆能力。

隐式失效可以作为项目进阶部分；基础版本先处理明确更新和冲突。

## 十、最终要回答的研究问题

### RQ1：版本感知是否能减少过期记忆误用？

比较普通向量检索和版本感知检索，观察：

- 过期记忆被检索的比例；
- 最终答案使用过期信息的比例；
- 当前状态问答准确率。

### RQ2：双视图能否兼顾当前与历史？

比较：

- 所有历史统一保存；
- 只保存当前状态；
- 当前状态＋历史版本双视图。

分别测试：

- 当前状态问题；
- 历史状态问题；
- 状态变化问题。

### RQ3：资源受限时方法是否仍然有效？

在不同存储和 token 预算下比较：

- 准确率；
- 过期错误率；
- token 消耗；
- 数据库存储大小。

### RQ4：额外管理开销是否可以接受？

测量：

- 记忆写入延迟；
- 更新关系判断延迟；
- 检索延迟；
- 峰值内存；
- 向量索引大小。

## 十二、对照方法

建议至少实现：

| 方法                                           | 作用                           |
| ---------------------------------------------- | ------------------------------ |
| Full History                                   | 完整历史上界，但成本高         |
| Recent Window                                  | 只保留最近若干轮               |
| Vector Top-K                                   | 最基本向量检索                 |
| Vector + Time                                  | 加时间衰减                     |
| [Mem0](https://arxiv.org/abs/2504.19413)-style | 原子抽取与动态更新基线         |
| Current State Only                             | 只保留当前状态                 |
| StateBudgetMem                                 | 你们的完整方法                 |
| Oracle Memory                                  | 人工提供正确记忆，作为理论上界 |

所有方法必须使用：

- 相同大模型；
- 相同向量模型；
- 相同问题；
- 相同 token 上限；
- 相同生成参数。

这样结果差异才能归因于记忆管理方法。

## 十三、评价指标

### 效果指标

- 最终回答准确率；
- 当前状态问题准确率；
- 历史问题准确率；
- Valid Recall@K；
- FAMA；
- 拒答准确率。

### 过期记忆指标

\[
\text{Stale Retrieval Rate}
=
\frac{\text{检索出的过期记忆}}
{\text{全部检索记忆}}
\]

\[
\text{Stale Usage Rate}
=
\frac{\text{使用过期信息导致的回答}}

\]

## 十四、实验顺序

### 实验一：验证版本管理

先不限制预算，只比较：

```text
Vector Top-K
Vector + Time
版本感知检索
```

证明版本关系能减少过期记忆使用。

### 实验二：验证双视图

比较：

```text
统一记忆库
仅当前状态
当前状态＋历史版本
```

证明双视图既能回答当前问题，又没有丢掉历史能力。

### 实验三：加入端侧预算

改变：

```text
存储预算：25%、50%、75%、100%
Token预算：128、256、512、1024
```

观察不同方法的准确率—成本关系。

### 实验四：状态变化压力测试

逐渐增加：

```text
历史长度
更新次数
冲突数量
临时失效事件
```

观察普通检索和你们方法何时开始失效。

### 实验五：端侧性能测试

在实际电脑 CPU、手机或开发板上测试：

```text
100条记忆
1,000条记忆
5,000条记忆
10,000条记忆
```

不一定需要在手机上运行完整大模型，重点测记忆管理模块。

## 十五、消融实验

必须分别去掉组件：

```text
完整方法

w/o Version
没有版本关系

w/o Dual View
当前和历史放在同一个库

w/o Query Type
不区分当前、历史和变化问题

w/o Stale Risk
检索时不惩罚过期风险

w/o Budget Awareness
不考虑token成本
```

这样才能说明提升究竟来自哪里，而不是只证明“整个系统碰巧更好”。

## 十六、最终研究产出

你们最终不是交付一个聊天应用，而是交付四类研究成果。

### 1. 一套方法

包含：

- 结构化原子记忆；
- 状态版本关系；
- 当前状态与历史版本双视图；
- 查询相关的有效性判断；
- 过期风险感知预算检索。

### 2. 一套可复现实验代码

能够自动完成：

```text
读取数据
→ 构建记忆
→ 运行检索
→ 生成答案
→ 计算指标
→ 输出图表
```

### 3. 一篇论文式报告

核心论证是：

1. 扁平向量记忆容易使用过期信息；
2. 版本管理可以减少这一问题；
3. 双视图可以兼顾当前与历史；
4. 在相同端侧预算下，方法具有更好的准确率—成本平衡。

### 4. 一个研究展示平台

展示的不是“聊天功能多强”，而是：

- 用户状态版本时间线；
- 普通检索与本方法的候选记忆对比；
- 哪条记忆当前有效；
- 是否使用了过期记忆；
- token、延迟和存储开销；
- 不同预算下的实验曲线。

## 附录 A：核心论文清单

以下论文按照它们在本研究路线中的作用分类。正式会议论文优先链接至会议页面；尚无正式会议页面的工作链接至 arXiv。

### A.1 长期记忆系统基础

1. **[MemoryBank: Enhancing Large Language Models with Long-Term Memory](https://ojs.aaai.org/index.php/AAAI/article/view/29946)**  
   提出面向大语言模型的长期记忆机制，使模型能够保存、召回和更新历史信息，并根据交互形成用户画像。论文引入受艾宾浩斯遗忘曲线启发的更新机制，根据时间与重要性选择性遗忘或强化记忆。  
   **本项目用途：** 支撑记忆写入、重要性、时间衰减和选择性遗忘。

2. **[Generative Agents: Interactive Simulacra of Human Behavior](https://dl.acm.org/doi/10.1145/3586183.3606763)**  
   提出“记忆流—检索—反思—规划”架构，检索时综合考虑语义相关性、时间新近度与重要性。  
   **本项目用途：** 支撑基础混合检索评分，并作为普通长期记忆方法的经典参考。

3. **[Mem0: Building Production-Ready AI Agents with Scalable Long-Term Memory](https://arxiv.org/abs/2504.19413)**  
   从连续对话中动态抽取、整合和检索关键记忆，减少完整历史带来的 token 与延迟开销；同时提出图结构版本以表达记忆间关系。  
   **本项目用途：** 支撑原子记忆抽取、去重、合并、更新和基础系统架构。

4. **[A-MEM: Agentic Memory for LLM Agents](https://openreview.net/forum?id=FiM0M8gcct)**  
   借鉴卡片盒笔记法，将记忆组织为带标签、关键词和动态连接的网络；新记忆可以建立关联并推动已有记忆演化。  
   **本项目用途：** 支撑替代、补充、冲突和版本关系的结构化表达。

### A.2 长期记忆数据集与评测

5. **[Evaluating Very Long-Term Conversational Memory of LLM Agents](https://aclanthology.org/2024.acl-long.747/)**（LoCoMo，ACL 2024）  
   构建跨多次会话的长期对话数据集，评测问答、事件总结、时间关系和多模态对话生成。  
   **本项目用途：** 测试长历史条件下的跨会话记忆和时间推理能力。

6. **[LongMemEval: Benchmarking Chat Assistants on Long-Term Interactive Memory](https://openreview.net/forum?id=pZiyCaVuti)**（ICLR 2025）  
   评测信息提取、跨会话推理、时间推理、知识更新和信息不足时拒答等长期记忆能力。  
   **本项目用途：** 测试版本更新、当前/历史查询和不确定状态处理。

7. **[Evaluating Memory in LLM Agents via Incremental Multi-Turn Interactions](https://arxiv.org/abs/2507.05257)**（MemoryAgentBench，ICLR 2026）  
   将记忆能力划分为准确检索、测试时学习、长程理解和选择性遗忘，强调记忆系统不仅要会保存，还要会更新和遗忘。  
   **本项目用途：** 支撑将“选择性遗忘”作为独立研究能力并设计相应评测。

8. **[From Recall to Forgetting: Benchmarking Long-Term Memory for Personalized Agents](https://arxiv.org/abs/2604.20006)**（Memora，2026）  
   面向持续数周至数月的个性化交互，重点测试记忆、推理和推荐，并提出 FAMA 指标，惩罚使用过期或失效记忆的行为。  
   **本项目用途：** 作为过期记忆误用、频繁状态更新和 FAMA 评测的核心依据。

### A.3 状态变化、冲突与时态一致性

9. **[STALE: Can LLM Agents Know When Their Memories Are No Longer Valid?](https://arxiv.org/abs/2605.06527)**（2026 预印本）  
   研究“隐式冲突”：后续事件没有直接否定旧记忆，却已经使旧状态失效；同时评测状态解析、过时前提抵抗和策略适应。  
   **本项目用途：** 支撑临时失效、隐式状态更新、状态传播和 `UNKNOWN_CURRENT` 设计。

10. **[MemConflict: Evaluating Long-Term Memory Systems Under Memory Conflicts](https://arxiv.org/abs/2605.20926)**（2026 预印本）  
    将记忆有效性定义为与查询相关的适用性问题，并构造动态、事实和条件冲突以及语义相似干扰项。  
    **本项目用途：** 支撑查询相关有效性、当前/历史双视图和冲突条件下的检索评测。

### A.4 端侧资源与预算约束

11. **[Forget to Improve: On-Device LLM-Agent Continual Learning via Budget-Curated Memory](https://arxiv.org/abs/2606.25115)**（2026 预印本）  
    直接研究端侧智能体在 RAM、能耗和通信预算受限时，如何依据单位字节净价值管理记忆的保留、分享与信任。  
    **本项目用途：** 支撑端侧存储预算、记忆淘汰、资源开销和真实设备实验。

12. **[Remember the Decision, Not the Description: A Rate-Distortion Framework for Agent Memory](https://arxiv.org/abs/2605.10870)**（DeMem，2026 预印本）  
    从决策导向的率失真角度研究有限预算下的记忆压缩：一条记忆是否值得保留，取决于遗忘后是否会改变未来决策。  
    **本项目用途：** 支撑预算约束下的记忆保留、压缩和准确率—成本权衡。
